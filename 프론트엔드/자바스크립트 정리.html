ECMAScript란?

ECMA(유럽 컴퓨터 제작자 협회. European Computer Manufacturers Association)는 정보통신기술, 전자제품을 위한 국제 표준기구다.
자바스크립트를 개발한 Netscapte가 더 향상된 표준화를 위해 기술 규격을 Ecma에 제출했고, 그에 따라 Ecma가 새롭게 제정한 표준이 바로 ECMA-262이다.
ECMAScript란 기술 규격에 따라 정의하고 있는 표준화된 스크립트 프로그래밍 언어를 말한다. ES라고도 한다.
발음할때는 보통 에크마스크립트라고 한다.
ECMAScript의 여러 버전을 '판'이라고 부른다.

자바스크립트의 구성 3요소

코어(ECMAScript), 문서객체모델(DOM), 브라우저객체모델(BOM)

  1. 코어(ECMAScript)
    문법, 타입, 선언문, 키워드, 예약어, 연산자, 객체 등의 언어의 저수준에 해당하는 부분을 정의한다.
    액션 스크립트, 플래쉬, NodeJS, JavaScript 등은 이러한 명세를 구현한 언어

  2. 문서 객체 모델(DOM : Document Object Model)
    XML을 HTML에서 사용할 수 있도록 확장한 애플리케이션 프로그래밍 인터페이스.
    전체 페이지를 노드의 계층 구조로 변환한다.

  3. 브라우저 객체 모델(BOM : Browser Object Model)
    BOM을 이용하면 브라우저에 표시된 페이지와는 별개의 컨텍스트에서 브라우저와 상호작용 할 수 있다.
    BOM을 주관하는 자바스크립트 표준이 존재하지 않는다는 문제가있었으나 HTML5가 BOM을 공식명세의 일부로 표준화하고 있기 때문에 크게 개선되었다.
    웹브라우저는 ECMAScript를 구현하는 호스트환경일뿐. 다른호스트환경으로 NodeJS, 어도비 플래시 등이 있다.

var와 let의 명확한 차이

참조
디테일차이는 "var와 let의 차이.js" 파일에서 보자.

  1. var는 중복선언 가능, let은 중복선언 불가능
  2. 호이스팅 매커니즘의 차이
    • var는 선언 호이스팅되고 undefined로 초기화 해준다.
    • let, const는 선언은 호이스팅 되지만 초기화가 되기전에 호출한다면 TDZ에 빠져서 ReferenceError : is not defined 에러
  3. scope차이
    • var는 함수레벨 스코프. [오로지 함수]의 코드 블록만을 지역 스코프로 인정한다.
      즉, 함수안에서 정의하면 지역변수가 되는것이고, 함수밖에있다면 전역변수가 되는것이다.
    • let, const는 블록레벨 스코프

scope

식별자의 유효범위를 뜻한다.
유효하다는것은 참조(접근) 할 수 있다는 뜻이다.
모든 식별자는 자신이 선언된 위치에 의해 다른코드가 자신을 참조할 수 있는 유효범위가 결정된다.
전역에 선언된 변수는 전역 스코프, 지역에 선언된 변수는 지역 스코프를 가진다.
거의모든 언어가 블록레벨{} 스코프를 따르는데
자바스크립트는 함수레벨 스코프를 따른다. 너무 불편함이 많아서 ECMA6에 도입된 let, const는 블록레벨 스코프를 사용할 수 있게 되었다.
만약 스코프라는 개념이 없으면 같은 이름을 갖는 식별자는 충돌을 일으키므로 프로그램 전체에서 하나밖에 사용 할 수 없다.

자바스크립트에서는 '전역'과 '지역' 2가지의 스코프가 있다.

스코프체인 (scope chain)

내부함수에서는 외부함수의 변수에 접근가능하지만, 외부함수에서는 내부함수의 변수에 접근할 수 없다.
이렇게 꼬리를 물고 계속 범위를 넓혀가며 찾는 관계를 스코프체인이라고 한다.

렉시컬 스코프 (Lexical scope)

이 변수가 사용가능한지 알기 위해(스코프) 소스코드 어디에서 선언되었는지를 고려한다는 것을 의미한다.

    var i = 10;   //i는 a함수안에서만 유효한놈이다.
    b();    //5출력.  b함수가 밑에있어도 호출할수있는건 호이스팅때문
}

function b(){
    console.log(i); //5출력!   b와 가까운 10이 아니다.  b함수안에서 i찾는데 없다. 그럼 b바깥에서 찾는다. 찾는 i는 전역변수 i
}

a();
TDZ(Temporal Dead Zone)

일시적인 사각지대. 변수가 '블록스코프의 시작지점'부터 '초기화가 시작되는 지점'까지의 구간.
let과 const는 TDZ의 영향을 받아 할당전에는 사용할 수 없게 된다.
이는 코드를 var보다 예측가능하게 하고 잠재적인 버그를 줄인다.

console.log(x);

              //브라우저에서는 x is not defined에러

실행컨텍스트 (Execution Context)

초급자는 알필요도, 알수도 없지만 중급자는 반드시 알아야 되는 필수개념.

  • scope, hoisting, this, function, closure 등의 동작원리를 담고있는 자바스크립트의 핵심원리.
  • 실행할수 있는 코드가 실행되기 위해 필요한 환경
  • 컨텍스트는 '문맥'이라는 말.
  • 현재 실행중인 컨텍스트에서 이 컨텍스트와 관련이 없는 코드가 실행되면 새로운 컨텍스트가 생성되고 스택에 쌓이면서 제어권이 이동한다.

호이스팅 (hoisting .끌어올림)

  • 자바스크립트 인터프리터가 변수나 함수의 선언을 스코프(유효범위) 최상위에 선언된 것 처럼 올리는 행위.
    (중요)초기화를 말하는게 아니다. 선언을 말하는것이다.
  • 즉, 선언과 초기화를 분리한 후 선언만 코드의 최상단으로 옮기는것을 말한다.
    그래서 선언은 됐는데 값은 할당되지 않은 undefined가 뜬다.
  • var는 함수레벨 스코프만 가지고,
    let, const는 블록 레벨 스코프(중괄호)를 가진다.
  • var는 호이스팅시 undefined로 변수의 값을 초기화하고
    let과 const는 호이스팅시 변수를 초기화하지 않는다.
  • 함수선언식은 호이스팅의 영향을 받고
    함수표현식은 호이스팅의 영향을 받지 않는다.

엄밀히 말하면 호이스팅이란
실행 컨텍스트 생성 시 렉시컬 스코프(어휘적 범위) 내의 선언이 끌어올려 지는 게 호이스팅이다
자바스크립트 엔진에서 런타임 이전에 먼저 함수를 선언한다.
반면에 함수표현식은 런타임 이전에 값을 undefined로 초기화만 시키고 런타임타이밍에 객체가 된다.
let과 const는 호이스팅이 되지 않는다고 명시된 경우가 많은데 잘못된 설명이다.
호이스팅이 안되는것이 아니라 TDZ때문에 안된것처럼 느껴지는 것이다.

콜백함수

다른함수에 인자로 넘겨지는 함수 (자바스크립트에서 함수는 일급객체이기 때문에 가능하다.)
콜백함수를 쓰는 쪽에서 인자로 전달받아 특정조건을 만족하면 그 때, 나중에 콜이 되는 함수이기 때문에 이름이 콜백함수.
예) 이때의 printTrue와 printFalse가 콜백함수.

function matchQuiz(answer, printTrue, printFalse){
    if(answer[0]==="무령왕릉"){
        printTrue();
    }else{
        printFalse();
    }
}

일반함수와 다른것 아니다. 그저, 호출방식에 의한 구분이다.
그러면 버튼에 클릭이벤트가 발생했을때 실행하는 함수도 콜백함수겠네? 응, 맞다.

DOM (Document Object Model)

문서객체 모델.

  • 자바스크립트에서 HTML이나 XML(Extensible Makrup Language)의 구조에 접근할수있는 표준.

  • DOM은 페이지를 노드의 계층 구조로 변환한다. EventTarget이라는 최상위 부모밑에 Node가 있고 Node밑에 Document와 Element가 있다. Element밑에 HTMLElement가 있다.
    document.getElementById("title") 이런식으로 썼던것은 document객체에 접근했던거고, 앞의 window를 생략할수 있었던 이유는 window는 전역객체이기 때문에 그렇다.
    또 최상위 요소인 EventTarget의 addEventListener를 써왔던 것이다.

  • DOM지원은 브라우저 제조사들의 우선순위 맨 위에 있으며 이런 노력의 댓가로 판올림 할 때마다 지원상황이 좋아지고 있다.
    DHTML(Dynamic HTML)을 지원하면서 개발자들은 웹페이지를 새로고침하지 않고도 콘텐츠의 모양을 바꿀수 있게 되었다. 그러나 넷스케이프와 MS가 개발한 DHTML은 완전히 달랐으므로 웹은 두 브라우저중 하나쪽만 지원하는 두 세력으로 갈라저 버릴것이라는 불안감이 팽배했다. 이때부터 웹통신 표준을 만들던 월드 와이드 웹 컨소시엄(W3C)에서 DOM 관련 작업을 하였다.
    무수히 많은 언어에서 DOM을 구현하지만 웹 브라우저에서 DOM은 ECMAScript를 통해 구현된다.

이벤트 버블링

한 요소에 이벤트가 발생하면 이 요소에 할당된 핸들러가 동작하고, 이어서 부모 요소의 핸들러가 동작하는것.
가장 최상단의 조상 요소를 만날때까지 이 과정이 반복되면서 요소 각각에 할당된 핸들러가 동작한다.
제일 깊은 곳에 위치한 요소의 이벤트가 부모 요소를 거슬러 올라가며 발생하는 모양이 마치 물속 거품과 닮았기 때문에 이런 이름이 붙었다.
자식에는 클릭이벤트를 안주고 부모에만 줘도 자식을 클릭했을대 부모의 이벤트가 발생한다.
부모와 자식에 이벤트를 주고 자식을 클릭하면 부모자식이 둘다 발생한다.
가장 안쪽의 요소는 타깃 요소라고 불리고 event.target 으로 접근할 수 있다. this 는 event.currentTaget을 의미한다.
버블링을 중단하려면 event.stopPropagation(); 또는 event.preventDefault(); 를 사용하면 된다. 그러나 꼭 필요한 경우를 제외하곤 버블링을 막지 않아야 한다. 아키텍처를 무너뜨리기 때문이다.

  • 태그는 이벤트버블링이 잘안되는데 검색해도 원인을 알 수 없다.

프로미스 ##### promise

  • 자바스크립트와 Node는 비동기 프로그래밍으로 이벤트 주도 방식을 활용하면서 콜백 함수를 많이 사용하게 된다. 콜백 함수 자체가 복잡한 것도 있고, 이해하기 어려운 자바스크립트 내용 중 하나이기도 하다.

  • 동기는 코드의 이해가 쉽다.

  • 이에 ES2015부터는 콜백 대신 API들이 프로미스 기반으로 재구성되고 있다. 따라서 프로미스에 대해 잘 이해하고 사용하게 된다면, 복잡한 콜백 함수의 지옥에서 벗어날 수 있으으며 훨씬 직관적으로 코딩할 수 있다.

  • 생성 : new Promise() 로 프로미스를 생성할 수 있다.
    생성자는 2개의 파라미터를 갖느다.
    첫번째는 성공했을때 실행할 콜백함수,
    두번째는 실패했을때 실행할 콜백함수.
    이름은 마음대로 해도상관없지만 관습적으로 resolve와 reject를 사용한다.

      인스턴스가 있다면 이제 `then과 catch` 메서드를 붙이는 것이 가능하다.
  resolve가 호출되면 then이 실행되고, reject가 호출되면 catch가 실행된다.
  즉, '.then'뒤에 비동기작업을 마치면 실행할 콜백함수들 써주면된다.

바닐라 js예제
var job1 = new Promise(function(resolve, reject){
  setTimeout(function(){
    resolve('resolved ok!');
  }, 2000);
});
job1.then(function(data){
  console.log('data', data);
})
그러나 보통은 아래와 같이 함수안에서 promise를 호출해준다.
function job1(){
  return new Promise(function(resolve, reject){
    setTimeout(function(){
      resolve('resolved ok!');
    }, 2000);
  });
}
job1().then(function(data){
  console.log('data', data);
})

  1. ? (optional chaining 연산자)
    구문법의 객체.속성 을 신문법에서 객체?.속성 이라는 문법이 생겼다.
    ?왼쪽이 null, undefined면 ?오른쪽 실행안하고 undefined를 반환한다.
    에러가 나면 밑의 문장들 다 실행안되기 때문에 조건이나 예외처리나 하지않고 간단하게 이문법으로 해결할 수 있다.
    사실 자바스크립트 엔진은 자동으로 찍어주고 있었다. 그런데 중첩 자료가 생기면 그때 에러가 난다.
    //유의. 자바스크립트에서 undefined나 null자체는 문제가 아니다. 기본자료형이잖아.
    그 놈의 하위를 .으로 접근하려고 하면 에러다.
    예) document.querySelector('#a')
    는 undefined라도 문제가 없지만
    그놈이 undefined일때 document.querySelector('#a').innerHTML은 에러다.
